Weex在iOS平台上官方Toast偶尔不显示问题

发表于 2018-05-18

遇到了什么问题？

  项目中要使用Weex就着手想把Weex官网上的示例用法自己建个新工程跑一跑看看效果，走到内置的modal组件的时候发现在我新建的iOS工程中运行的时候该出现toast的时候不出现，但是在安卓上正常，官方给的demo工程中运行正常。

检查自己写的代码

  Weex代码是复制的官方示例:
let modal = weex.requireModule('modal')
modal.toast({
    message: 'This is a toast',
    duration: 0.3
})

  既然js代码最终都转化为了iOS、安卓平台本地的代码，那么我检查iOS的试图层吧，看看到底有没有我的toast UI，于是我这么做了把toast的时间增大到1000：

let modal = weex.requireModule('modal')
modal.toast({
    message: 'This is a toast',
    duration: 1000
})

  同样的一个界面先看下正常时候的UI层级，红色箭头指向的地方就是toast展示的地方：

  接着看不正常的时候的UI层级，红圈圈出的位置没有出现本应该出现的toast：

查原因

  网上查资料好像没有，只能跟源码，找到sdk中的WXModalUIModule.m，toast是一个试图，我要看下这个试图的父试图是哪位，我想给其父试图设置颜色查看这个父试图的真实面目，我找到如下位置：

WXAssertMainThread();
UIView *superView =  [[[UIApplication sharedApplication] windows] objectAtIndex:0];
if (!superView) {
    superView =  self.weexInstance.rootView;
}

  找到了这个superView，给它来个红色背景：

WXAssertMainThread();
UIView *superView =  [[[UIApplication sharedApplication] windows] objectAtIndex:0];
if (!superView) {
    superView =  self.weexInstance.rootView;
}
superView.backgroundColor = [UIColor redColor];

  看正常时候的UI层级图，红色superView出现（其实是废话，正常当然出现了，不正常的时候难道就不出现，没有吗）：

  不正常的时候我就不贴图了，是看不到红色的superView的。

toast所在的superView不见了？

  这个superView到底是什么？继续看源码：

UIView *superView =  [[[UIApplication sharedApplication] windows] objectAtIndex:0];

  是一个window，window当然也是一个view，源码中取的下标为0的window作为superView，而出现问题的时候我给superView添加了一个红色背景，检查试图层级没看到次红色superView，果真丢了？我要把所有此时的window都打印出来看看：

  执行如下语句：

po [[UIApplication sharedApplication] windows]

  正常的情况下结果如下：

<__NSArrayM 0x604000258630>(
<UIWindow: 0x7febe521ad80; frame = (0 0; 375 667); autoresize = W+H; gestureRecognizers = <NSArray: 0x600000259b00>; layer = <UIWindowLayer: 0x600000221b80>>,
<WXWindow: 0x7febe5368bf0; baseClass = UIWindow; frame = (110 0; 60 40); gestureRecognizers = <NSArray: 0x604000250f20>; layer = <UIWindowLayer: 0x604000035e20>>
)

  不正常的情况下结果如下：

<__NSArrayM 0x604000056620>(
<UIWindow: 0x7f8ed460d010; frame = (0 0; 375 667); hidden = YES; gestureRecognizers = <NSArray: 0x608000242940>; layer = <UIWindowLayer: 0x6080000263c0>>,
<UIWindow: 0x7f8ed4416880; frame = (0 0; 375 667); autoresize = W+H; gestureRecognizers = <NSArray: 0x604000455f00>; layer = <UIWindowLayer: 0x604000421fe0>>,
<WXWindow: 0x7f8ed4675400; baseClass = UIWindow; frame = (110 0; 60 40); gestureRecognizers = <NSArray: 0x608000448130>; layer = <UIWindowLayer: 0x60800002c7e0>>
)

  虽然此时我还不知道toast不出现的具体原因，但是我此时兴奋的时候二者结果有差异，这是好现象，继续追查差异性的东西，不正常的时候打印的结果多了一个window，其中在结果下标为0的那个window的hidden属性为yes，yes？那么我刚才设置了背景色就是给这货设置的，因为它为hidden所以看不到？

有点苗头了

  先不要关闭不正常的时候运行的iOS程序，我们尝试下运行时给这个下标为0的window设置hidden = NO，执行如下代码：

(lldb) po [[UIApplication sharedApplication] windows]
<__NSArrayM 0x60400045e900>(
<UIWindow: 0x7fa975d09730; frame = (0 0; 375 667); hidden = YES; gestureRecognizers = <NSArray: 0x60400005bd80>; layer = <UIWindowLayer: 0x604000221cc0>>,
<UIWindow: 0x7fa975d147d0; frame = (0 0; 375 667); autoresize = W+H; gestureRecognizers = <NSArray: 0x604000241950>; layer = <UIWindowLayer: 0x604000225840>>,
<WXWindow: 0x7fa975c5cd80; baseClass = UIWindow; frame = (110 0; 60 40); gestureRecognizers = <NSArray: 0x6000002578b0>; layer = <UIWindowLayer: 0x600000423e80>>
)

(lldb) e [(UIView *)0x7fa975d09730 setHidden:NO]
(lldb) caflush

  执行完上面的debug代码之后，再查看UI层级发现toast出现了，toast的superView也出现了，红色背景也看到了，这说明出现问题的时候该存在的试图还是存在的只不过hidden了，看不到。

为什么会多出来一个window？

  其实这个问题倒是很好说的，我们新建iOS工程的时候会有一个默认的东西看图：

  也会有一个默认的配置看图：

  当然往往我们可能会在AppDelegate中自己初始化根vc，类似这样：

self.window = [[UIWindow alloc] initWithFrame:[[UIScreen mainScreen] bounds]];
WeexBaseController *mainVc = [[WeexBaseController alloc] initWithSourceUrl:weexRenderUrl];
[self.window setRootViewController:mainVc];
[self.window makeKeyAndVisible];
return YES;

  之后可能就是顺其自然的编码该干啥干啥。如果这样，一般情况下不会有问题，二般情况下就是现在的情况下就会有问题，我们自己初始化了window之后如果不把系统开始默认加载的那个Main.storyboard从配置中移除掉那么系统默认的window还是存在的只不过展示了我们自己初始化的window，隐藏了开始默认的window。当然聪明的你会习惯性的移除Main.storyboard的配置也就不会出现类似的问题了。

后记

  阿里的工程师把下标为0的window作为toast的superView而不是keyWindow，不是太理解。问题虽然很低级但是追问题的过程还是让我收获不少的。
